## [2. Snakes (USACO Gold)](https://usaco.org/index.php?page=viewproblem2&cpid=945)
**Тема:** DP, prefix sums, running max

**Ключові ідеї:**
- Для фіксованого сегмента `l..r` оптимальний розмір сітки `s` **не вибирається**, а **визначається умовою**: `s = max(d[l..r])` (інваріант сегмента)
- Вартість сегмента: `cost(l,r) = s * len - sum(l,r)`, де `sum(l,r)` через prefix sums

**Де помилився в мисленні:**
- Ти “попав” рівно там, де часто ламається мислення: ти намагався оптимізувати всередині сегмента через перебір `s`, хоча `s` не є незалежним вибором — він інваріант сегмента (дорівнює максимуму на відрізку). Тобто ти додав третю вісь, яка насправді визначається даними, а не “керується рішенням”.

**Патерн для майбутнього:**
- Якщо параметр має **покрити всі елементи сегмента** (`s ≥ a[i]`), то оптимум часто `s = max` → **не додавай його як стан/перебір**
- Для `max(l..r)` у сегментному DP: фіксуй `l` і розширюй `r`, тримаючи running max: `mx = max(mx, a[r])`

## [4. Team Building (USACO Gold)](https://usaco.org/index.php?page=viewproblem2&cpid=673)
**Тема:** DP по парах, 2D prefix sums

**Ключові ідеї:**
- Інваріант **E (exact-last):** `dp[i][j][k]` = # способів, де **остання** пара рівно \((i,j)\).
- Перехід: `dp[i][j][k] = 1[b[j] < a[i]] * ps[i-1][j-1][k-1]`, де `ps` — 2D префікс сум `dp` для шару.
- Відповідь: `ans = sum dp[i][j][K]`

**Де помилився:**
- Змішав “префіксний dp” і “exact-last dp” → база/перехід/ans описували різні речі.
- Спочатку брав лише `dp[i-1][j-1]` замість суми по прямокутнику → потрібен `ps`.
- Зламав 1-based індекси сортуванням разом із `a[0]=0`.
- Переплутав шар у `ps` (будував `ps[k]` через сусідів `k-1`) і місцями робив `break`, залишаючи “дірки”.

**Патерн для майбутнього:**
- Спочатку зафіксувати **один** інваріант dp і тримати його всюди.
- Якщо перехід = **сума по 2D прямокутнику**, одразу думати про **2D prefix sums**.

## [5. Stamp Painting (USACO Gold)](https://usaco.org/index.php?cpid=791&page=viewproblem2)
Тема: DP по run-length, доповнення (total - bad), sliding window / queue

Ключові ідеї:
- Рахувати доповнення: `good = m^n - bad`, де `bad` — рядки без блоку ≥ k однакових підряд
- Prefix-state: важлива лише довжина хвоста однакових (run), обрізана до `k-1`
- “Bad → Good” стається тільки коли хвіст має `k-1` однакових і додаємо той самий колір (1 варіант)
- Хвости — це конвеєр, тому віднімання робиться через чергу з лагом `k-2`

Помилки:
- Замість “існує блок довжини k десь” я рахував “блок k на початку/в кінці” → пропускав випадки з блоком всередині
- Намагався “забороняти другу появу блоку” через страх дубліката, хоча кожен рядок у префіксному DP рахується рівно один раз
- Зламав інваріант черги: пушив не ту величину / переплутав, який елемент треба віднімати (зсув на 1) + офф-бай-ван у циклі довжини
  
Патерн:
- Для “є підряд k” → рахуй `bad` через run-length і роби `good = total - bad`; якщо стани зсуваються, тримай їх через queue/deque

## [6. Coin Combinations II (CSES 1636)](https://cses.fi/problemset/task/1636/)
Тема: DP на монетах, комбінації без врахування порядку, 1D DP

Ключові ідеї:
- `dp[s]` = кількість способів набрати суму `s`, використовуючи лише ті типи монет, які вже пройшли у зовнішньому циклі
- Зовнішній цикл по монетах підтримує префікс дозволених типів: після обробки `j`-тої монети в `dp` вже накопичена повна відповідь для всіх сум з монет `c[0..j]`
- Тобто `dp` — це не “остання монета” і не “довжина”, а саме відповідь для префікса набору монет
- Достатньо одновимірного масиву `dp`
- Той самий `dp` можна оновлювати inplace: прохід по сумі зліва направо дозволяє використовувати поточну монету багато разів
- Саме порядок циклів задає правильний інваріант: зовнішній цикл по монетах не дає рахувати різні порядки як різні способи

Помилки:
- Спочатку зробив стан `dp[sum][last]` і думав через “останню монету”
- Не влучив у правильний інваріант: тут треба тримати не останній вибір, а відповідь по префіксу дозволених монет
- Через це роздув пам’ять і вперся в RE / MLE
- Не одразу побачив, що вся задача фактично тримається на сенсі зовнішнього циклу

Патерн:
- Якщо треба рахувати комбінації без порядку, стан часто означає “відповідь після обробки префікса типів”
- Зовнішній цикл тоді накопичує відповідь лише по дозволених монетах, і це автоматично прибирає перестановки
- Якщо перехід додає поточний тип необмежену кількість разів, часто вистачає 1D DP з проходом по сумі зліва направо

## [7. Coding Company(CSES 1665)]
**Дата:** 2026-03-10
**Тема:** DP по відсортованому масиву, незавершені об'єкти, counting DP

**Ключові ідеї:**
- відсортувати skill-и
- дивитись на процес зліва направо, а не на готові команди
- стан задається не конкретними складами команд, а кількістю **відкритих** команд
- відкрита команда = уже має мінімум, але ще не зафіксований максимум
- при переході між сусідніми skill-ами кожна відкрита команда однаково додає `diff` до штрафу
- тому для майбутнього важливо знати не склад відкритих команд, а лише їх кількість
- фінальний валідний стан: усі люди оброблені, всі команди закриті

**Що було неочевидно:**
- я дивився на задачу як на розбиття людей по конкретних командах
- через це стан виходив занадто “детальним” і незручним
- правильний погляд з'являється, якщо перейти від статичної структури до процесу побудови
- треба було помітити не “хто в якій команді”, а “що лишилось незавершеним під час проходу”

**Інваріант стану:**
- після обробки перших `i` людей кожен із них уже належить рівно одній команді
- `j` означає лише кількість команд, які вже почались, але ще не закриті
- штраф у стані вже враховує всі пройдені проміжки між skill-ами

**Локальна структура переходу:**
- між сусідніми значеннями skill усі відкриті команди симетричні
- якщо відкрито `j` команд, то перехід через `diff` додає однаковий внесок `j * diff`
- після цього новий програміст:
  - відкриває нову команду
  - закриває одну з відкритих
  - входить в одну з відкритих і лишає її відкритою
  - або утворює одноосібну команду
- саме звідси виникають кратності переходів через `j`

**Де була головна складність:**
- не у формулах
- не у переходах
- а в пошуку правильної агрегованої інформації для стану

**Патерн для майбутнього:**
- не питати одразу “який тут DP?”
- спочатку питати:
  - **Що лишилось незавершеним під час проходу?**
  - **Яка найменша агрегована інформація достатня, щоб коректно продовжити процес?**
- якщо багато об'єктів поводяться однаково, часто треба зберігати не їх identities, а лише їх кількість
- якщо відповідь будується під час sweep/scan по відсортованому масиву, хороший стан часто описує саме “відкриті” об'єкти

**Мій meta-висновок:**
- вузьке місце було не у відповіді на правильні питання, а в тому, що я сам їх не поставив

## [8. Round Subset (Codeforces 837D)](https://codeforces.com/contest/837/problem/D)
**Тема:** DP, trailing zeros, вибір стану

**Ключові ідеї:**
- кількість trailing zeros визначається кількістю пар `(2, 5)` у добутку
- тому кожне число зручно замінити на `(cnt2, cnt5)`
- DP: `dp[kk][pw]` = максимум двійок, якщо вже вибрано `kk` елементів і набрано `pw` п’ятірок
- у відповідь іде `max(min(dp[k][pw], pw))`
- стан краще індексувати по п’ятірках, бо їх верхня межа менша, ніж у двійок

**Помилки:**
- спочатку не побачив правильну модель відповіді: не було ідеї, як рахувати trailing zeros через пари `(2, 5)`
- після цього хотів будувати стан по двійках, а не по п’ятірках, що роздуває простір станів
- неправильно уявляв префіксне DP: думав, що з позиції `i` треба ще перебирати наступний `j`, хоча для нового елемента є лише два варіанти — взяти або не взяти

**Патерн для майбутнього:**
- якщо в задачі є trailing zeros, майже завжди треба думати не про саме число, а про факторизацію по `2` і `5`
- у DP з двома “ресурсами” одну величину часто вигідно тримати як індекс, а другу — як значення
- у префіксному DP треба кожен раз перевіряти, чи справді потрібен “наступний вибір”, чи стан уже все це покриває через “взяти / не взяти”

## [9. Why Did the Cow Cross the Road II (USACO Gold)](https://usaco.org/index.php?page=viewproblem2&cpid=721)
Тема: DP на двох префіксах, non-crossing matching, інваріант state

Ключові ідеї:
  * `dp[i][j]` — це оптимум на префіксах `a[1..i]` і `b[1..j]`, а не “відповідь, що закінчується в `(i,j)`”
  * Для prefix-DP треба пропагувати оптимум по більшому прямокутнику: пропустити елемент зліва / справа
  * Match `(i,j)` — це лише ще один варіант переходу, якщо пара сумісна
  * Є монотонність по включенню префіксів: більший прямокутник уже містить оптимум меншого

Помилки:
  * Я фактично мислив state як “беремо саме `(i,j)`”, але transition-и не підтримували цю семантику
  * Не пропагував already-found optimum вперед, тому рахував лише match-діагоналі
  * Через це `dp[i][j]` не означав оптимум на префіксах, а був напівзламаним “ending-state”

Патерн:
  * У DP по двох послідовностях спочатку жорстко зафіксувати семантику state: “оптимум на префіксах” чи “оптимум, що закінчується тут”
  * Кожен перехід має зберігати зміст state; якщо state = prefix optimum, то зазвичай є переходи “skip left / skip up / take pair”

# [10. Minimal Grid Path (CSES 3359)] 
**Тема:** лексикографічний мінімум, діагоналі, відсікання станів

## Ключові ідеї

- Діагональ `d` — це всі клітинки з `i + j = d`, тобто всі стани після однакової кількості ходів.
- На одній діагоналі всі шляхи мають однакову довжину, тому їх можна порівнювати лексикографічно як префікси.
- Для діагоналі достатньо зберігати лише ті клітинки, до яких можна дійти **мінімальним префіксом** серед усіх шляхів до цієї діагоналі.
- Якщо клітинка на цій діагоналі має строго гірший префікс, її можна одразу відкинути: далі вона вже не стане кращою.
- Якщо кілька клітинок мають один і той самий мінімальний префікс, треба лишати всі: їхні продовження можуть відрізнятися.
- Перехід на наступну діагональ:
  - беремо всіх сусідів збережених клітинок,
  - знаходимо мінімальну букву серед них,
  - лишаємо тільки тих сусідів, у яких ця буква.
- Через це стан — не `best string` для клітинки, а **множина клітинок поточної діагоналі**.

## Інваріант

- На діагоналі `d` зберігаються всі й тільки ті клітинки, до яких можна дійти рядком, що є мінімальним серед усіх шляхів довжини `d + 1`.

## Де затупив

- Зберігав цілий рядок у стані.
- Не врахував, що `string + char` і `min(string, string)` роблять перехід дорогим.
- Не побачив, що тут відповідь треба будувати пошарово, а не тягнути рядок для кожної клітинки окремо.

## Патерн

- Якщо треба знайти лексикографічно мінімальний шлях, часто вигідно:
  - йти по шарах / рівнях,
  - зберігати всі позиції, сумісні з мінімальним префіксом,
  - на кожному кроці фільтрувати тільки по мінімальному наступному символу.

## Складність

- `O(n^2)`, бо кожна діагональ обробляється окремо, без копіювання повних рядків.


## USACO Palindromic Paths

Спочатку не бачив цілісну модель задачі. Хотілось притягнути “звичайний dp”, але тут ключ не в одному шляху, а в двох синхронних фронтах.

### Ідея
Паліндром на шляху означає, що символи мають збігатися попарно з двох боків, тому треба рухатись **назустіч від початку і кінця**.  
Стан зберігає не весь шлях, а лише **дві позиції фронтів**.  
На фіксованому шарі достатньо тримати `(r1, r2)`, бо `c1, c2` відновлюються з діагоналей.  
Перехід = 4 комбінації рухів двох фронтів.

### Що було складно
Проблема була не в техніці DP, а в тому, що я не одразу розпізнав правильну модель.  
Коли умова зв’язує початок і кінець об’єкта, треба думати не про prefix/range state, а про **pair state / two-front DP**.

### Висновок
Якщо в задачі є паліндром, симетрія або попарна відповідність далеких позицій, треба рано перевіряти питання:  
**“чи не треба будувати об’єкт одночасно з двох боків?”**


## Cow Jog (USACO) https://usaco.org/index.php?page=viewproblem2&cpid=496

**Дата:** 2026-03-24  
**Тема:** мінімальне покриття підпослідовностями, greedy on tails, dual sequence

**Ключові ідеї:**
- Для кожної корови корисно дивитись на `p = x + vT`.
- Одна смуга відповідає **строго зростаючій** підпослідовності `p`.
- Тому задача = розбити `p` на мінімальну кількість строго зростаючих підпослідовностей.
- Двоїста форма: відповідь дорівнює довжині **найдовшої незростаючої підпослідовності**.
- Greedy-версія: зберігати хвости відкритих підпослідовностей і класти `p` у підпослідовність з найбільшим хвостом, який **строго менший** за `p`.
- Якщо такого хвоста нема, треба відкривати нову підпослідовність.
- Реалізаційно це можна бачити двома способами:
  - `multiset` хвостів;
  - patience/LNIS-style масив `dp`.

**Де помилився / що було тонким:**
- Спочатку сплутав strictness:
  - не **неспадні** підпослідовності,
  - а саме **строго зростаючі**.
- Через це в коді випадок рівності оброблявся неправильно.

**Патерн для майбутнього:**
- Якщо задача виглядає як  
  **“розбити послідовність на мінімальну кількість монотонних підпослідовностей”**,  
  то треба одразу питати:
  - яка підпослідовність є **несумісною** всередині одного ланцюжка?
- Часто відповідь = максимальна довжина саме такої “доповняльної” підпослідовності.

## LCIS (Codeforces 10D) https://codeforces.com/problemset/problem/10/D

**Дата:** 2026-03-26  
**Тема:** DP, running maximum

**Ключові ідеї:**
- Стан: `dp[j]` — довжина найкращої LCIS, що закінчується в `b[j]`.
- Новий кінець у `b[j]` можна створити лише коли `a[i] == b[j]`.
- Наївно: перебирати всі `k < j` з `b[k] < b[j]` → `O(n*m*m)`.
- Оптимізація: при `a[i] == b[j]` умова `b[k] < b[j]` стає `b[k] < a[i]`.
- Для фіксованого `i` можна йти по `j` зліва направо і тримати `mx = max(dp[k])` серед уже пройдених `k`, де `b[k] < a[i]`.
- Тоді:
  - `b[j] < a[i]` → оновлюємо `mx`
  - `b[j] == a[i]` → пробуємо `dp[j] = max(dp[j], mx + 1)`

**Де стопився:**
- Не бачив, чому важлива умова `<` / `>`.
- Не клеївся перехід від `b[k] < b[j]` до `b[k] < a[i]`.
- Editorial став зрозумілим лише після явного `O(n*m*m)`.

**Патерн для майбутнього:**
- Якщо є перехід “максимум по всіх `k < j`”, спершу побудувати прозорий повільний DP.
- Потім перевірити, чи умову на `k` можна прив’язати до константи зовнішнього циклу і замінити перебір на running maximum.

## Зеник і щастя https://algotester.com/en/ArchiveProblem/DisplayWithEditor/81483

**Дата:** 2026-03-30  
**Тема:** DP на сітці, критична колонка, префікс/суфікс максимуми

### Ідея
- Після `dp` і відновлення всіх оптимальних шляхів через `good` задача звужується до випадку `SZ(full) == 1`.
- Нехай `c = full[0]`. Тоді кожен шлях проходить колонку `c` як суцільний вертикальний відрізок:
  - зайшов у рядку `x`,
  - вийшов вправо в рядку `y`,
  - `x <= y`.

### Що треба порахувати в `else`
- Для кожного рядка `r`:
  - найкращий шлях, що **проходить через** `(r, c)`;
  - найкращий шлях, що **обходить** `(r, c)` зверху;
  - найкращий шлях, що **обходить** `(r, c)` знизу.

### Підготовка
- `suf[i][j]` — максимум від `(i, j)` до `(n, m)`.
- `ps[i]` — префіксна сума по колонці `c`.
- `enter[i]` — найкраще зайти в колонку `c` на рядку `i`:
  - `dp[i][c - 1]`, або спецвипадок для `c == 1`.
- `leave[i]` — найкраще вийти з колонки `c` на рядку `i`:
  - `suf[i][c + 1]`, або спецвипадок для `c == m`.

### Перепис суми шляху
- Якщо шлях у колонці `c` проходить відрізок `[x..y]`, його сума:
  - `enter[x] + (ps[y] - ps[x - 1]) + leave[y]`
- Перегрупування:
  - `A[x] = enter[x] - ps[x - 1]`
  - `B[y] = leave[y] + ps[y]`
- Тоді сума шляху через `[x..y]` = `A[x] + B[y]`.

### Агрегація максимумів
- `mxA[i]` — максимум `A[x]` для `x <= i`
  - найкраще зайти в колонку і ще не опуститися нижче `i`.
- `mxB[i]` — максимум `B[y]` для `y >= i`
  - найкраще вийти з колонки, якщо дійшли до `i` або нижче.
- `prefBest[i]` — найкращий шлях, який у колонці `c` **вийшов вправо не нижче `i`**
  - тобто обхід рядка `r` зверху дає `prefBest[r - 1]`.
- `sufBest[i]` — найкращий шлях, який у колонці `c` **зайшов не вище `i`**
  - тобто обхід рядка `r` знизу дає `sufBest[r + 1]`.

### Формула для рядка `r`
- Через `r`:
  - `mxA[r] + mxB[r] - a[r][c]`
- Обхід зверху:
  - `prefBest[r - 1]`
- Обхід знизу:
  - `sufBest[r + 1]`

Отже:

```cpp
ans = max({ans, mxA[r] + mxB[r] - a[r][c], prefBest[r - 1], sufBest[r + 1]});
```

## USACO 1089 https://usaco.org/index.php?page=viewproblem2&cpid=1089

**Дата:** 2026-04-07  
**Тема:** subset DP, порядок символів, стиснення рядка до переходів

### Ключова ідея

Задача насправді не про симуляцію співу, а про вибір **порядку літер**.  
Для фіксованого порядку новий `hum` потрібен рівно тоді, коли сусідній перехід у рядку йде **назад** відносно цього порядку.

Тому рядок треба стиснути до `cnt[a][b]` = кількість **сусідніх** переходів `a -> b`.

### Правильна модель

Будуємо порядок літер зліва направо.

`dp[mask]` = мінімальна кількість bad-переходів, уже порахованих для всіх порядків, у яких префікс складається рівно з літер `mask`.

Перехід:
беремо `bit` як **останню** літеру поточного префікса `mask`.

Тоді новий внесок — усі переходи `j -> bit`, де `j` ще **не** в `mask`.  
Чому: у фінальному порядку всі такі `j` стоятимуть правіше за `bit`, отже `j -> bit` буде backward.

### Що було важливо побачити

1. Достатньо дивитись тільки на **сусідні** пари в рядку.  
   Глобальна коректність блоку випливає з транзитивності порядку.

2. Пари `x -> x` не залежать від порядку взагалі.  
   Якщо дві однакові літери стоять **сусідньо**, між ними обов’язково новий `hum`.  
   Тому це окремий константний внесок `ans`.

3. DP рахує тільки переходи між **різними** літерами, які залежать від вибраного порядку.

### Реалізаційний інваріант

У твоєму коді:

- `ans` = кількість сусідніх пар `a[i] == a[i+1]`
- `cnt[x][y]` рахується тільки для `x != y`
- `dp[mask]` добудовується з `pMask = mask ^ (1 << bit)`
- `fine = sum(cnt[j][bit])` по всіх `j`, яких нема в `mask`

Це означає: кожен backward-перехід рахується **рівно один раз** — у момент, коли його права літера вперше входить у префікс.

### Де була головна пастка

Найнеприємніше місце задачі:  
легко подумати, що “повторення літери десь у рядку” саме по собі створює окрему складність.

Насправді константні лише **сусідні** `x -> x`.  
Наприклад, в `abab` немає константного внеску, хоча `a` і `b` повторюються.

Ще одна пастка — приховане обмеження через scoring/subtasks:  
bitmask DP проходить, бо реально треба працювати лише з різними літерами, що з’явилися в рядку.

### Патерн для майбутнього

Якщо задача має вигляд:

- треба вибрати порядок маленької множини;
- довгий рядок можна стиснути до локальних переходів;
- вартість залежить від того, хто стоїть раніше/пізніше,

то варто думати в такому ланцюжку:

**рядок → частоти сусідніх переходів → порядок елементів → subset DP по префіксу порядку**

### Складність

`O(|s| + 2^k * k^2)`, де `k` — кількість різних літер у рядку.


## Making Friends — https://usaco.org/index.php?page=viewproblem2&cpid=1499

**Дата:** 2026-04-08  
**Тема:** complete multipartite graph, complement as union of cliques, subset DP, mask precomputation

### Ключові ідеї

- Умова еквівалентна тому, що фінальний граф має бути **complete multipartite**.
- Тобто вершини треба розбити на групи:
  - всередині групи ребер немає,
  - між різними групами всі ребра є.
- Еквівалентно: у **доповненні** граф має бути **об’єднанням клік**.

### Правильна модель

- `dp[mask]` — мінімальна кількість змін, щоб підграф на вершинах `mask` став коректним.
- Перебираємо `submask ⊆ mask` як одну з частин розбиття.
- Локальна ціна:
  - видалити всі ребра всередині `submask`,
  - додати всі відсутні ребра між `submask` і `mask \ submask`.

### Корисна статистика

- `inside[mask]` — кількість **наявних** ребер всередині `mask`.
- Дуже корисний патерн: після передобчислення за `O(2^n)` кількість ребер **у будь-якій підмножині** можна діставати за `O(1)`.
- Передобчислення:
  - беремо один встановлений біт `v`,
  - `pmask = mask \ {v}`,
  - новий внесок = кількість сусідів `v` у `pmask`.

Інваріант:
- старі внутрішні ребра вже враховані в `inside[pmask]`,
- нові — лише ті, що інцидентні доданій вершині.

### Бітові техніки

- Якщо зберігати суміжні вершини `v` як бітову маску `adj[v]`, то кількість сусідів вершини `v` всередині підмаски `mask` можна знайти як:

`popcount(adj[v] & mask)`

- Це дуже корисний патерн: перетин через `&`, кількість встановлених бітів через `popcount`.

- `__builtin_ctz(mask)` дає індекс **наймолодшого встановленого біта**.
- Це зручно, щоб вибрати одну вершину `v` для переходу від `mask` до `pmask`.

### Оптимізація

Кількість **наявних** ребер між двома неперетинними множинами:

`inside[A ∪ B] - inside[A] - inside[B]`

Тоді кількість **відсутніх** ребер між ними:

`|A| * |B| - (inside[A ∪ B] - inside[A] - inside[B])`

Це прибирає зайвий множник `n` у переході.

### Патерн для майбутнього

Якщо в subset DP з’являється “кількість ребер між двома частинами”, то перша думка:
- чи можна виразити її через
  - `inside[union]`
  - `inside[left]`
  - `inside[right]`

### Ключова ідея

Гравцю важливо не **хто** з ним у команді, а лише **розмір** його команди.

Тому треба моделювати не саму підмножину, а `x = |A|`.

---

### Редукція

Для фіксованого `x` кожен гравець має 4 типи:

- `onlyA`
- `onlyB`
- `both`
- `none`

Тоді:

- `onlyA` уже в `A`
- `onlyB` уже в `B`
- з `both` треба добрати `x - onlyA`
- якщо є `none`, такий `x` неможливий

Умова існування:

- `none = 0`
- `onlyA <= x`
- `onlyA + both >= x`

Кількість способів для цього `x`:

- вибрати `x - onlyA` із `both`

Для гравця допустимі розміри — `[L_i, R_i]`.

Тоді:

- для `A`: `x ∈ [L_i, R_i]`
- для `B`: `x ∈ [n - R_i, n - L_i]`

Якщо умова залежить не від складу групи, а лише від її **агрегату** (тут — розміру), то часто треба перебирати саме цей агрегат, а не групу.
